## P0 Finding: Installable NLS Loader Missing

### Mechanism (verified, not speculated)

1. The bundled `main.js` calls `localize(<number>, fallback)` at module-init time. The bundler rewrites source `localize('key', 'message')` calls into `localize(<index>, 'message')`. At runtime, `lookupMessage(index)` reads `globalThis._VSCODE_NLS_MESSAGES[index]`. If missing, it **throws `NLS MISSING: <index>`**.
2. The NLS data IS present and intact ‚Äî `out/nls.messages.js` has **18,903 entries, zero non-string holes**, and entry 138 is exactly `"Extensions"` (the `PolicyCategory.Extensions` label from `src/vs/base/common/policy.ts:33`). Verified by copying the file to TEMP and `require()`-ing it.
3. **`out/main.js` never imports `out/nls.messages.js`.** `grep -c "nls.messages" out/main.js` = 5 (just literal string references, no `import` or `require`). `package.json` has `"main": "./out/main.js"`, `"type": "module"`. Electron loads `main.js` as the ESM entry ‚Äî and **nothing populates `_VSCODE_NLS_MESSAGES` before `update.config.contribution.ts` (imported by `main.ts`) hits its first `localize(138, ...)` call**.
4. Dev mode works because the AMD-loader path resolves NLS keys differently (string-based, not index-based) and has its own `nls.ts` initializer.

### Blast radius

- `dist/test/GoatIDE-Test-Setup-x64.exe` ‚Äî **CONFIRMED BROKEN** (manually reproduced this session)
- `dist/GoatIDE-Setup-x64.exe` (GA build) ‚Äî **LIKELY BROKEN** (same build session 2026-05-16/17, same gulp + electron-builder pipeline, same `package.json.main` shape, same `out/main.js` + standalone `out/nls.messages.js` layout)

### Hypothesis on root cause

The upstream microsoft/vscode 1.117.0 build pipeline likely generates a **bootstrap wrapper** that loads `nls.messages.js` before importing `out/main.js`, or alternatively prepends/inlines `nls.messages.js` into `main.js` at the start. Our gulp+electron-builder pipeline ships `nls.messages.js` as a peer file of `main.js` but doesn't wire the load order. Two non-mutually-exclusive root causes:

1. **The fork dropped or skipped a bootstrap script** (e.g., `out/bootstrap-main.js` or `out/cli.js`) that should have been the `package.json` `main` entry instead of `out/main.js`. Note: `out/cli.js` IS present in the bundle but `package.json.main` points to `main.js` directly.
2. **A gulp task transformation step is missing** ‚Äî e.g., a step that prepends `import './nls.messages.js';` to `main.js` at packaging time.

Investigation should compare our gulp + bundling output against an upstream microsoft/vscode v1.117.0 reference Windows .zip (downloadable from update.code.visualstudio.com ‚Äî *if* you're willing to grab one for diagnostic comparison only).

## ADDENDUM: Fix Applied (2026-05-19, same day)

**P0 is RESOLVED in source.** Root cause turned out to be different from the initial NLS-loader hypothesis ‚Äî and simpler than expected.

### Actual root cause

`src/main.ts:25-29` had a Plan 12-06 heuristic that defaulted `process.env['VSCODE_DEV'] = '1'` whenever `import.meta.dirname` contained `'out'` but NOT `'.asar'`. The comment justified this with: *"Production builds package out/ inside resources/app.asar/"*. In practice the v2.1 installable pipeline ships `resources/app/out/` **unpacked** (only `resources/app/node_modules.asar` is asar-archived; the app dir itself is plain), so the heuristic incorrectly fired against production installs.

The downstream chain that caused the visible crash:
1. Heuristic sets `process.env['VSCODE_DEV'] = '1'` in installable mode (wrong)
2. `startup()` calls `bootstrapESM()` ‚Üí `setupNLS()`
3. `setupNLS()` first line: `if (process.env["VSCODE_DEV"] || !messagesFile) return void 0;` ‚Äî early-returns because VSCODE_DEV is set
4. `_VSCODE_NLS_MESSAGES` is never populated
5. First `localize(<number>, fallback)` call (from `policy.ts` ‚Üí `update.config.contribution.ts`) throws `NLS MISSING: 138`

Diagnosis path: I added three `console.error()` traces to the installable's bundled `main.js` (`doSetupNLS START`, file-read marker, `EARLY RETURN` marker), relaunched, and observed `VSCODE_DEV=true` at the `EARLY RETURN` site. The NLS data IS correct; the loader IS correct; the heuristic that gates the loader was wrong.

### Fix

Replaced the asar-marker heuristic with a `src/`-sibling existence check (`src/main.ts:8-32`). A dev checkout always has `<entryDir>/../src` because that's where the TypeScript sources gulp builds from. Installable builds ship only `out/` ‚Äî there is no `src/` sibling. This signal is deterministic and does not depend on the pipeline's asar-packing behaviour.

```ts
import * as path from 'node:path';
import * as fs from 'original-fs';

if (typeof process.env['VSCODE_DEV'] === 'undefined' || process.env['VSCODE_DEV'] === '') {
	const entryDir = import.meta.dirname;
	const srcSibling = path.join(entryDir, '..', 'src');
	if (entryDir.includes('out') && fs.existsSync(srcSibling)) {
		process.env['VSCODE_DEV'] = '1';
	}
}
```

Both `path` and `original-fs` imports moved to the top of the file (no semantic change ‚Äî ESM hoists imports ‚Äî but reads cleaner). `npm run compile-check-ts-native` confirms the change typechecks clean.

### Diagnosis chain (preserved for future regression debugging)

The original UAT report theorized "NLS loader is missing entirely". The actual root cause was subtler ‚Äî the loader IS present, the data IS correct, but a separate dev-mode-detection heuristic incorrectly fired against production builds, which gated the loader behind a `process.env["VSCODE_DEV"]` early-return. The fix path: add three `console.error()` traces into `doSetupNLS()` of the installed bundle's `main.js` ‚Üí relaunch ‚Üí observe `VSCODE_DEV=true messagesFile=<correct path>` at the `EARLY RETURN` site. That immediately reframed the search from "find the loader" to "find what's setting `VSCODE_DEV=1`".

The lesson: when a localize-by-index call fails in a bundled main process, check what's setting `process.env['VSCODE_DEV']` before assuming the NLS data is missing.

### What this UAT does NOT yet validate

- **Full rebuild cycle** ‚Äî the installable on disk is hot-patched, not freshly built. A `bash scripts/package-goatide.sh --test` + reinstall + re-UAT is the canonical end-to-end test. Slot before the next versioned release.
- **DecisionNode authoring END-TO-END** ‚Äî the command is in the palette (AUTH-01 surface confirmed) but completing the full flow (anchor selection ‚Üí rationale text ‚Üí kernel write ‚Üí graph display) requires kernel-side anchor data this UAT did not seed. Unit tests for AUTH-01 cover the bridge `KernelClient.createDecisionNode` method.
- **Multi-folder workspace cross-repo edges** (XREPO-03) ‚Äî UAT used a single folder; the cross-repo *graceful degradation* was verified, but the cross-repo *activation* path needs a multi-folder workspace.
- **`update#downloaded` Mandate D dialog** ‚Äî verified by unit tests (`update-downloaded restart triggers quitAndInstall` + `update-downloaded later does NOT trigger quitAndInstall` both PASS), but live UAT requires publishing a real GitHub Release with a higher version than the installed binary.
