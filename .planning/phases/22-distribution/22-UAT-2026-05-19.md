## P0 Finding: Installable NLS Loader Missing

### Mechanism (verified, not speculated)

1. The bundled `main.js` calls `localize(<number>, fallback)` at module-init time. The bundler rewrites source `localize('key', 'message')` calls into `localize(<index>, 'message')`. At runtime, `lookupMessage(index)` reads `globalThis._VSCODE_NLS_MESSAGES[index]`. If missing, it **throws `NLS MISSING: <index>`**.
2. The NLS data IS present and intact — `out/nls.messages.js` has **18,903 entries, zero non-string holes**, and entry 138 is exactly `"Extensions"` (the `PolicyCategory.Extensions` label from `src/vs/base/common/policy.ts:33`). Verified by copying the file to TEMP and `require()`-ing it.
3. **`out/main.js` never imports `out/nls.messages.js`.** `grep -c "nls.messages" out/main.js` = 5 (just literal string references, no `import` or `require`). `package.json` has `"main": "./out/main.js"`, `"type": "module"`. Electron loads `main.js` as the ESM entry — and **nothing populates `_VSCODE_NLS_MESSAGES` before `update.config.contribution.ts` (imported by `main.ts`) hits its first `localize(138, ...)` call**.
4. Dev mode works because the AMD-loader path resolves NLS keys differently (string-based, not index-based) and has its own `nls.ts` initializer.

### Blast radius

- `dist/test/GoatIDE-Test-Setup-x64.exe` — **CONFIRMED BROKEN** (manually reproduced this session)
- `dist/GoatIDE-Setup-x64.exe` (GA build) — **LIKELY BROKEN** (same build session 2026-05-16/17, same gulp + electron-builder pipeline, same `package.json.main` shape, same `out/main.js` + standalone `out/nls.messages.js` layout)
- `../VSCode-win32-x64/GoatIDE.exe` (gulp portable app) — **LIKELY BROKEN** (verified same `package.json.main = ./out/main.js`, `"type": "module"`, same `out/` directory structure with separate `nls.messages.js`)
- macOS `.dmg` / Linux builds — **LIKELY BROKEN** (same upstream gulp pipeline; not testable from Windows host)

### Hypothesis on root cause

The upstream microsoft/vscode 1.117.0 build pipeline likely generates a **bootstrap wrapper** that loads `nls.messages.js` before importing `out/main.js`, or alternatively prepends/inlines `nls.messages.js` into `main.js` at the start. Our gulp+electron-builder pipeline ships `nls.messages.js` as a peer file of `main.js` but doesn't wire the load order. Two non-mutually-exclusive root causes:

1. **The fork dropped or skipped a bootstrap script** (e.g., `out/bootstrap-main.js` or `out/cli.js`) that should have been the `package.json` `main` entry instead of `out/main.js`. Note: `out/cli.js` IS present in the bundle but `package.json.main` points to `main.js` directly.
2. **A gulp task transformation step is missing** — e.g., a step that prepends `import './nls.messages.js';` to `main.js` at packaging time.

Investigation should compare our gulp + bundling output against an upstream microsoft/vscode v1.117.0 reference Windows .zip (downloadable from update.code.visualstudio.com — *if* you're willing to grab one for diagnostic comparison only).

## ADDENDUM: Fix Applied (2026-05-19, same day)

**P0 is RESOLVED in source.** Root cause turned out to be different from the initial NLS-loader hypothesis — and simpler than expected.

### Actual root cause

`src/main.ts:25-29` had a Plan 12-06 heuristic that defaulted `process.env['VSCODE_DEV'] = '1'` whenever `import.meta.dirname` contained `'out'` but NOT `'.asar'`. The comment justified this with: *"Production builds package out/ inside resources/app.asar/"*. In practice the v2.1 installable pipeline ships `resources/app/out/` **unpacked** (only `resources/app/node_modules.asar` is asar-archived; the app dir itself is plain), so the heuristic incorrectly fired against production installs.

The downstream chain that caused the visible crash:
1. Heuristic sets `process.env['VSCODE_DEV'] = '1'` in installable mode (wrong)
2. `startup()` calls `bootstrapESM()` → `setupNLS()`
3. `setupNLS()` first line: `if (process.env["VSCODE_DEV"] || !messagesFile) return void 0;` — early-returns because VSCODE_DEV is set
4. `_VSCODE_NLS_MESSAGES` is never populated
5. First `localize(<number>, fallback)` call (from `policy.ts` → `update.config.contribution.ts`) throws `NLS MISSING: 138`

Diagnosis path: I added three `console.error()` traces to the installable's bundled `main.js` (`doSetupNLS START`, file-read marker, `EARLY RETURN` marker), relaunched, and observed `VSCODE_DEV=true` at the `EARLY RETURN` site. The NLS data IS correct; the loader IS correct; the heuristic that gates the loader was wrong.

### Fix

Replaced the asar-marker heuristic with a `src/`-sibling existence check (`src/main.ts:8-32`). A dev checkout always has `<entryDir>/../src` because that's where the TypeScript sources gulp builds from. Installable builds ship only `out/` — there is no `src/` sibling. This signal is deterministic and does not depend on the pipeline's asar-packing behaviour.

```ts
import * as path from 'node:path';
import * as fs from 'original-fs';

if (typeof process.env['VSCODE_DEV'] === 'undefined' || process.env['VSCODE_DEV'] === '') {
	const entryDir = import.meta.dirname;
	const srcSibling = path.join(entryDir, '..', 'src');
	if (entryDir.includes('out') && fs.existsSync(srcSibling)) {
		process.env['VSCODE_DEV'] = '1';
	}
}
```

Both `path` and `original-fs` imports moved to the top of the file (no semantic change — ESM hoists imports — but reads cleaner). `npm run compile-check-ts-native` confirms the change typechecks clean.
